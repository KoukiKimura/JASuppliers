# 仕入システム バッチ 詳細設計書

**プロジェクト名**: JASuppliers 仕入システム
**作成日**: 2026年5月14日
**最終更新**: 2026年5月14日
**バージョン**: 1.0
**対象要件**: C-004、C-006、C-008、CC-005、CC-006、CC-007、CC-010、B-006、B-007、B-009、共通基盤

---

## 1. 目的

POS売上取込、販売集計、委託精算、消化仕入確定、支払データ出力、会計連携CSV出力など、初期リリースで必要な非同期・一括処理を定義する。初期リリースではWindowsローカルMock確認環境を前提に手動実行を基本とし、後続で定時実行へ拡張できる構成にする。

## 2. 対象バッチ

| バッチID | 名称 | 主な関連要件 | 実行契機 |
|---|---|---|---|
| POS_IMPORT | POS売上取込 | C-004、CC-005、B-006 | 画面から手動実行 |
| SALES_AGGREGATION | 販売集計 | C-004、CC-006、B-007 | 取込後または画面から手動実行 |
| CONSIGNMENT_SETTLEMENT | 委託精算作成 | C-006、C-008 | 締め期間指定で手動実行 |
| CONSUMPTION_PURCHASE_CONFIRM | 消化仕入確定 | CC-007、CC-010 | 締め期間指定で手動実行 |
| PAYMENT_EXPORT | 支払データ出力 | C-008、CC-010、B-009 | 精算確定後に手動実行 |
| ACCOUNTING_EXPORT | 会計連携CSV出力 | C-007、CC-010、B-009 | 対象期間指定で手動実行 |
| AUDIT_LOG_ARCHIVE | 操作ログ保全 | 共通 | 管理者が手動実行。定時化候補 |

## 3. テーブル定義

本書で定義する業務テーブルには、特記がない限り `created_at`、`updated_at`、`created_by`、`updated_by` を保持する。実行履歴は追記型で管理し、削除しない。

### 3.1 job_execution_logs

| カラム | 型 | 必須 | 説明 |
|---|---|---|---|
| id | bigserial | ○ | 主キー |
| job_id | varchar(50) | ○ | バッチID |
| status | varchar(20) | ○ | queued、running、success、failed |
| requested_by | bigint |  | 実行ユーザーID |
| parameters | jsonb |  | 実行パラメータ |
| started_at | timestamp |  | 開始日時 |
| finished_at | timestamp |  | 終了日時 |
| target_count | integer | ○ | 対象件数 |
| success_count | integer | ○ | 成功件数 |
| error_count | integer | ○ | エラー件数 |
| error_summary | text |  | エラー概要 |
| created_by | bigint |  | 作成者ID |
| updated_by | bigint |  | 更新者ID |
| created_at | timestamp | ○ | 作成日時 |
| updated_at | timestamp | ○ | 更新日時 |

### 3.2 failed_jobs

Laravel標準のキュー失敗管理テーブルを利用する。キューを使わない手動実行バッチでも、予期しない例外は `job_execution_logs.error_summary` に記録する。

## 4. API仕様

| API | Method | 概要 | 主な権限 |
|---|---|---|---|
| `/api/admin/jobs` | GET | 実行履歴一覧取得 | job.view |
| `/api/admin/jobs/{job_id}/run` | POST | バッチ手動実行 | job.run |
| `/api/admin/jobs/{execution_id}` | GET | 実行結果詳細取得 | job.view |

### 4.1 実行リクエスト例

```json
{
  "parameters": {
    "from_date": "2026-04-01",
    "to_date": "2026-04-30",
    "supplier_id": 10
  }
}
```

### 4.2 実行結果レスポンス例

```json
{
  "execution_id": 101,
  "job_id": "CONSIGNMENT_SETTLEMENT",
  "status": "success",
  "target_count": 120,
  "success_count": 120,
  "error_count": 0
}
```

## 5. 画面仕様

| 画面ID | 画面名 | 主な項目 | 主な操作 |
|---|---|---|---|
| JOB-001 | バッチ実行一覧 | バッチID、名称、最終実行日時、最終結果 | 実行、履歴表示 |
| JOB-002 | バッチ実行履歴 | 実行日時、実行者、パラメータ、件数、結果 | 検索、詳細表示 |
| JOB-003 | バッチ実行結果 | 成功件数、エラー件数、エラー概要 | 再実行、関連画面遷移 |

## 6. 共通処理仕様

1. 実行権限を確認する。
2. 実行パラメータを検証する。
3. `job_execution_logs` に `running` で開始記録を作成する。
4. 対象データを抽出する。
5. バッチ固有処理を実行する。
6. 成功件数とエラー件数を集計する。
7. 全体結果を `success` または `failed` で記録する。
8. 登録、確定、出力を伴う処理は `audit_logs` に記録する。

## 7. バッチ別処理仕様

| バッチID | 主処理 | 成果物 |
|---|---|---|
| POS_IMPORT | POS売上CSVを検証し、取込バッチ、取込明細、取込エラーへ登録する | `pos_import_batches`、`pos_sales_lines`、`pos_import_errors` |
| SALES_AGGREGATION | POS取込明細を商品・出荷者・店舗・日付単位で集計する | `pos_sales_lines` の集計結果 |
| CONSIGNMENT_SETTLEMENT | 委託販売集計から精算明細を作成する | `settlements`、`settlement_lines` |
| CONSUMPTION_PURCHASE_CONFIRM | 販売実績を仕入確定データへ反映する | `settlements`、`settlement_lines` |
| PAYMENT_EXPORT | 確定済み精算を対象に全銀協形式CSVを作成する | `bank_transfer_exports`、`payment_batches` |
| ACCOUNTING_EXPORT | 確定済み仕入・精算データから会計連携CSVを作成する | `accounting_exports`、`document_files` |
| AUDIT_LOG_ARCHIVE | 操作ログを期間指定で抽出し、保全ファイルを作成する | `document_files` |

## 8. バリデーション

| 対象 | ルール |
|---|---|
| job_id | 定義済みバッチIDであること |
| from_date、to_date | 日付形式、開始日が終了日以前 |
| supplier_id | 指定時は存在する出荷者ID |
| POS_IMPORT | 取込ファイルがCSV形式であり、ヘッダがPOS連携詳細設計書の定義と一致すること |
| PAYMENT_EXPORT | 対象精算が確定済みであり、支払データ未出力であること |
| ACCOUNTING_EXPORT | 対象データが確定済みであること |

## 9. エラーハンドリング

| 区分 | 扱い |
|---|---|
| 入力不備 | 実行前にHTTP 422を返却し、履歴は作成しない |
| 行単位エラー | 可能な範囲で正常行を処理し、エラー件数と概要を記録する |
| 業務整合性エラー | 該当データを処理対象外にし、エラー概要へ記録する |
| システム例外 | トランザクションをロールバックし、`failed` として記録する |

## 10. テスト観点

| 区分 | 観点 |
|---|---|
| Unit | バッチパラメータ検証、対象データ抽出、件数集計 |
| Feature | 手動実行API、権限不足403、実行履歴作成、失敗時履歴更新 |
| Feature | POS取込、販売集計、精算作成、消化仕入確定、支払データ出力、会計連携CSV出力 |
| E2E | Windowsローカル環境でのCSV配置、手動実行、出力ファイル確認 |
| Regression | 同一条件での重複実行防止、確定済みデータの再出力制御 |
