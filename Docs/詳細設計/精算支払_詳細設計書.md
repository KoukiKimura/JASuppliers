# 精算支払 詳細設計書

**プロジェクト名**: JASuppliers 仕入システム
**作成日**: 2026年5月14日
**最終更新**: 2026年5月14日
**バージョン**: 1.0
**対象要件**: C-006、C-008、CC-007、CC-010、B-009

---

## 1. 目的

委託仕入、委託消化仕入、買取仕入で共通利用する精算、支払予定、支払確定、全銀協形式CSV出力の詳細仕様を定義する。

---

## 2. 対象範囲

| 区分 | 対象 |
|------|------|
| 対象精算 | 委託精算、委託消化仕入確定、買取支払 |
| 支払方式 | 銀行振込 |
| 出力形式 | 全銀協形式CSV |
| 精算サイクル | 初期リリースは月次 |
| 対象外 | 週次精算、銀行API送信、会計ソフト別専用支払連携 |

---

## 3. 要件対応

| 機能ID | 詳細設計での対応 |
|--------|------------------|
| C-006 | `settlements` / `settlement_lines` による委託精算 |
| C-008 | `payment_batches` / `payment_lines`、全銀協形式CSV出力 |
| CC-007 | 委託消化仕入確定の精算ヘッダ・明細作成 |
| CC-010 | 委託消化支払確定、支払明細管理 |
| B-009 | 買取仕入の支払予定・支払確定 |

---

## 4. テーブル定義

本書で定義する業務テーブルには、特記がない限り `created_at`、`updated_at`、`created_by`、`updated_by` を保持する。精算・支払確定後の履歴性を優先するため、確定済みデータは削除せず取消ステータスで管理する。

## 7. 支払確定処理

1. 精算締め済みデータを対象に支払予定を抽出する。
2. 支払先の標準口座を取得する。
3. 口座未登録または無効口座をエラーにする。
4. `payment_batches` と `payment_lines` を作成する。
5. 経理担当者が支払内容を確認する。
6. 支払バッチを確定する。
7. 全銀協形式CSVを出力し、出力履歴を記録する。

---

## 8. バリデーション

| 対象 | ルール |
|------|--------|
| 精算締め | 未解決取込エラー、未照合請求、未承認調整がないこと |
| 支払バッチ | 同一支払日の未取消バッチ重複を警告 |
| 支払先 | 有効な振込口座が必須 |
| 支払額 | 0円以下は出力対象外。ただし控除明細として保持する |
| 取消 | 締め後取消は管理者権限と理由入力が必須 |

---

## 9. テスト観点

- 委託、委託消化、買取の精算が同じ支払バッチへ集約できること。
- 口座未登録の支払先がエラーになること。
- 支払確定後に全銀協形式CSVが出力されること。
- 確定済み支払バッチが直接編集できないこと。
- 取消時に理由と監査ログが残ること。
