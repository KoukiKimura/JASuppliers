## 7. リスクと対応方針

| リスク | 内容 | 対応 |
|--------|------|------|
| POS仕様未確定 | 実POSのデータ形式が不明 | 初期は想定CSVで実装し、ステージング取込で差分吸収する |
| 3形態同時実装 | 初期リリース範囲が広い | 共通マスタ・POS取込・精算基盤を先に作り、各形態の差分を段階的に実装する |
| 金額計算の差異 | 税、値下げ、返品、端数処理で誤差が出る | 基本設計で計算単位、丸め、締め後訂正を明文化する |
| ローカル環境差異 | 確認者ごとにWindows環境が異なる | Laravel Herd + PostgreSQL Windows版を標準手順とし、Seederで同一データを再現する |
| 出荷者ポータル後回し | 生産者向け確認が初期リリースに含まれない | 初期は管理者・担当者画面で精算明細を確認できるようにする |
| 複数店舗化 | 初期から複数店舗業務を作り込むと範囲が膨らむ | 初期は単店舗運用とし、店舗コード保持により拡張余地を確保する |
| フロントエンド未確定 | 実装時にBlade、Vue、Reactの選択が揺れる | Phase 1の最初に構成を確定し、基本設計とAGENT.mdへ反映してから画面実装を進める |
| 依存ライブラリのパッチ差異 | PHP、Laravel、PostgreSQLのパッチ差異でローカル挙動が変わる | メジャー方針は固定し、各環境では同一メジャー内の最新パッチを優先する |

## 9. 実装開始時の最初の作業

Phase 1では、以下の順序で着手する。

1. フロントエンド構成を確定し、基本設計とAGENT.mdへ反映する。
2. Laravel 13.xの本体を追加する。
3. `.env.example`、PostgreSQL接続、アプリキー、基本設定を整える。
4. 認証・権限の土台を作る。
5. 共通レイアウト、ナビゲーション、ダッシュボード最小版を作る。
6. 最小のMigration / Seeder / Featureテストを追加し、ローカル起動手順で確認する。
