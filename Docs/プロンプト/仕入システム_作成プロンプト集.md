# 仕入システム 作成プロンプト集

**プロジェクト名**: JASuppliers 仕入システム  
**作成日**: 2026年5月14日  
**最終更新**: 2026年5月14日  
**バージョン**: 1.3

---

## 1. 目的

本書は、JASuppliers 仕入システムでここまで作成した資料・サンプルデータ・運用方針を、別セッションまたは別環境で再現できるようにするためのプロンプト集である。

単なる会話ログではなく、同じ作成物を作成するために必要な前提、実行順、期待成果物、レビュー観点を整理している。

---

## 2. 使用方法

### 2.1 新しいAIセッションで再現する場合

1. GitHubリポジトリ `https://github.com/KoukiKimura/JASuppliers.git` を取得する。
2. `AGENT.md` を最初に読ませる。
3. 本書の「4. 実行順プロンプト」を上から順番に実行する。
4. 各プロンプト実行後、生成物と既存資料の整合性を確認する。
5. 最後に `AGENT.md` と本書を更新し、Gitへコミット・pushする。

### 2.2 既存資料を更新する場合

1. 今回の追加依頼に該当するプロンプトを「4. 実行順プロンプト」へ追記する。
2. 既存資料への影響を確認する。
3. 変更した資料、`AGENT.md`、本書を更新する。
4. 作成物が再現できる粒度で、期待成果物・レビュー観点を補足する。

## 3. 共通前提

| 項目 | 前提 |
|------|------|
| システム名 | JASuppliers 仕入管理システム |
| 対象業務 | 産直市場・農産物直売所における仕入業務の一元管理 |
| 仕入形態 | 委託 / 委託消化 / 買取 の3形態 |
| 初期リリース範囲 | 3形態すべてを実装対象とする |
| 出荷者ポータル | 後続フェーズ |
| POS連携 | 既存データ形式サンプルがないため、想定CSV形式で設計しサンプルデータを作成する |
| Mock確認 | Windowsローカル環境で構築・実行する方式を主とする |
| デプロイ | 初期開発後に別途検討する |
| 技術方針 | PHP 8.5 / Laravel 13.x / PostgreSQL 18.x を初期標準とする |
| DB切替 | Laravelの `.env` / `config/database.php` により将来切替可能にする |
| Docker | WSL2前提のため必須にしない |
| ローカル実行 | Laravel Herd + PostgreSQL Windows版を標準候補とする |
| ソース管理 | GitHub `https://github.com/KoukiKimura/JASuppliers.git` |

## 5. 再実行用統合プロンプト

既存のやり取りを一括で再現したい場合は、以下を新しいAIセッションで使用する。

```text
あなたはJASuppliers 仕入システムの設計支援AIです。
まずAGENT.mdを読み、既存のフォルダ構成、技術方針、初期リリース範囲、Git運用方針を把握してください。

目的は、汎用的な仕入システムを作成し、各要件によりカスタマイズしていくためのひな形を作成することです。
対象業務は産直市場・農産物直売所の仕入管理です。
仕入形態は、委託、委託消化、買取の3形態を初期リリースで全て対象にしてください。

技術方針は、PHP 8.5、Laravel 13.x、PostgreSQL 18.xを初期標準とします。
DBはLaravelの.env/config/database.phpで将来切替可能な設計にしてください。
WSL2/Docker Desktopは必須にせず、WindowsローカルではLaravel Herd + PostgreSQL Windows版を標準候補にしてください。
Mock確認は、相手側を含む各ローカル環境で構築・実行する方法を主とします。
Web公開・クラウド・VPS等へのデプロイ方式は後日検討としてください。

POS連携の既存データ形式サンプルはありません。
販売、返品、値下げ、委託、委託消化、買取を含む想定CSV仕様を定義し、サンプルCSVを作成してください。

出荷者ポータルは後続フェーズとし、初期リリースでは管理者・仕入担当者・店舗スタッフ・経理担当者向け機能を優先してください。
```

## 7. 今後の更新ルール

- プロンプトを実行して成果物を作成・更新した場合は、必ず本書を更新する。
- 追加したプロンプトは「4. 実行順プロンプト」に追記する。
- 追記時は、入力プロンプト、期待成果物、レビュー観点をセットで記載する。
- 既存プロンプトの前提が変わった場合は、「3. 共通前提」と「5. 再実行用統合プロンプト」も更新する。
- フォルダ構成や資料状態が変わった場合は、`AGENT.md` も更新する。
- 更新後は、要件定義・ロードマップ・基本設計・本書の間に矛盾がないか確認する。
