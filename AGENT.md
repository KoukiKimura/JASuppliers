## 1. プロジェクト概要

- **システム名**: JASuppliers 仕入管理システム
- **対象業務**: 産直市場・農産物直売所における仕入業務の一元管理
- **仕入形態**: 委託 / 委託消化 / 買取 の3形態
- **参考モデル**: JAあぐりタウン げんきの郷（愛知県大府市）
- **ソース管理**: GitHub `https://github.com/KoukiKimura/JASuppliers.git`
- **配布方針**: 各環境ではGitHubから最新ソースを取得し、ローカル環境で実行する

## 3. 設計上の重要な前提

| 項目 | 内容 |
|------|------|
| 仕入形態 | 委託（C）・委託消化（CC）・買取（B）の3形態を区別して管理する |
| 初期リリース範囲 | 委託・委託消化・買取の3形態すべてを実装対象とする |
| 店舗範囲 | 初期リリースは単店舗運用。店舗コードを保持し、将来の複数店舗化に備える |
| 機能ID体系 | 委託: C-001〜、委託消化: CC-001〜、買取: B-001〜、共通: X-001〜 |
| POS連携 | 既存サンプルがないためCSV形式を仮定し、`Work/サンプルデータ/POS取引明細_サンプル.csv` を基準データとする |
| 消費税 | 軽減税率（8%）と標準税率（10%）を区分して管理する |
| 精算サイクル | 初期リリースは月次精算。週次精算は後続フェーズで検討する |
| 出荷者ポータル | 後続フェーズで対応する。初期リリースでは管理者・担当者向け機能を優先する |
| 監査ログ | 全操作ログを記録し、7年間保管する（法令要件） |
| 文字コード | UTF-8 |
| 日付形式 | YYYY-MM-DD（ISO 8601） |

## 4.1 Git運用・環境反映方針

- 公式リポジトリは `https://github.com/KoukiKimura/JASuppliers.git` とする。
- 他環境への反映は、原則としてGitHubから最新ソースを取得して各環境でローカル実行する。
- WSL2 / Docker Desktop は必須条件にしない。
- Windowsローカルでは Laravel Herd、Composer、PostgreSQL Windows版を使う構成を標準手順とする。
- Mock確認は、相手側を含む各ローカル環境で構築・実行する方法を主とする。
- Web公開・クラウド・VPS等へのデプロイ方式は、Laravel実装後に別途検討する。
- ローカル実行手順は `Work/Readme.md` を参照する。
- 機密情報、`.env`、DBダンプ、生成物、依存パッケージはGit管理しない。

## 5. 作業上の注意点

### 5.1 ドキュメント作成・更新時の注意

- **要件定義書を起点にすること**  
  設計書の内容は必ず `Docs/要件定義/仕入システム_要件定義書.md` の機能IDと対応させる。  
  要件定義書にない機能を設計書に追加してはならない。

- **設計書の作成順序を守ること**  
  要件定義 → 基本設計 → 詳細設計の順に進める。  
  前工程のドキュメントが確定していない状態で次工程を開始しない。

- **既存ドキュメントと整合性を確認すること**  
  新しいドキュメントを作成・更新する際は、関連する既存ドキュメントと矛盾がないかを確認する。

- **このファイル（AGENT.md）を最初に更新すること**  
  フォルダ構成やドキュメント追加があった場合は、作業完了後に必ず本ファイルの「フォルダ構成」セクションを更新する。

- **プロンプト集を更新すること**  
  プロンプトを実行して成果物を作成・更新した場合は、作業完了前に `Docs/プロンプト/仕入システム_作成プロンプト集.md` へ入力プロンプト、期待成果物、レビュー観点を追記・整理する。  
  目的は、同じプロンプト群を使用して同じ作成物を再作成できる状態を維持することである。

### 5.2 AIエージェントへの指示時の注意

- 要件定義書を参照させる場合は `#file:Docs/要件定義/仕入システム_要件定義書.md` を明示する。
- 設計フェーズが変わる場合は **新しいチャットセッション** を開始してコンテキスト汚染を防ぐ。
- AIが生成したコード・設計書は必ず人間がレビューし、ハルシネーションがないか確認する。
- 個人情報・接続文字列・APIキーなどの機密情報は絶対にAIに渡さない。
- 詳細な手順は `Work/AI設計書生成ガイド.md` を参照すること。
- ローカル実行手順や環境構築手順は `Work/Readme.md` を参照すること。

### 5.3 データベース設計時の注意

- テーブル定義は必ず第3正規形以上とする。
- 全テーブルに `created_at`・`updated_at`・`created_by`・`updated_by` カラムを設ける。
- 論理削除カラム（`deleted_at` または `is_deleted`）を設ける。
- 外部キー制約を必ず定義する。
- DDL・マイグレーションはPostgreSQLを初期標準として作成する。
- 将来的なDB切替を考慮し、Laravel Migration / Query Builder / Eloquentを優先する。
