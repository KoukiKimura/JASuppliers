## 3. 今回発生した長時間停止の原因と回避

### 3.1 原因

`psql` をパスワード未指定で実行したため、PowerShell上でパスワード入力待ちになった。

この状態ではプロンプトが見えにくく、コマンドが停止しているように見える。今回の中断前に50分以上待機した原因はこれである。

### 3.2 回避ルール

PostgreSQLへコマンド接続する場合は、必ず以下を使う。

| 指定 | 目的 |
|------|------|
| `$env:PGPASSWORD='...'` | パスワード入力待ちを避ける |
| `-w` | パスワード入力プロンプトを禁止する |
| `connect_timeout=3` | 接続待ちを短時間で打ち切る |
| `timeout_ms` | Codex側のコマンド実行待ちを制限する |

管理者ユーザー `postgres` で接続する場合も同じルールを適用する。パスワード入力を対話で待つ形式の `psql` は使わない。

安全な確認コマンド例:

```powershell
$pgBin = "$env:LOCALAPPDATA\PostgreSQL\18-server\bin"
$env:PGPASSWORD = "jasuppliers_dev_password"
& "$pgBin\psql.exe" -w "host=127.0.0.1 port=5432 dbname=jasuppliers_dev user=jasuppliers_user connect_timeout=3" -c "SELECT current_database(), current_user;"
```

避けるコマンド:

```powershell
psql -h 127.0.0.1 -U postgres -d postgres
```

上記はパスワード入力待ちで止まる可能性がある。

## 4. 導入手順

### 4.1 PHP / Composer / Laravel Installer

PHPは公式 `php.new` が使用する Herd Lite の配布物をユーザー領域へ配置する。

配置先:

```text

```

確認:

```powershell

```

期待例:

```text
PHP 8.5.0
Composer version 2.8.12
Laravel Installer 5.27.0
```

### 4.2 Node.js / npm

既に導入済み。

確認:

```powershell
node --version
npm --version
```

### 4.3 PostgreSQL

PostgreSQLはWindows版インストーラーで導入する。

導入後、以下のパスを利用する。

```text
%LOCALAPPDATA%\PostgreSQL\18-server\bin
```

確認:

```powershell
psql --version
Get-Service postgresql-x64-18
```

### 4.4 PATH

ユーザー環境変数 `Path` は、PostgreSQL / PHP系として以下のみを使う。

```text
%LOCALAPPDATA%\PostgreSQL\18-server\bin
%USERPROFILE%\.config\herd-lite\bin
```

旧 `%LOCALAPPDATA%\PostgreSQL\18.3\bin` は一時導入時の残存パスであり、2026年5月15日の再レビューで削除済み。別環境で残っている場合は、`psql` の参照先が曖昧になるため削除する。

PowerShellを開き直すと反映される。

現在のPowerShellだけで確認する場合:

```powershell
$env:Path = "$env:LOCALAPPDATA\PostgreSQL\18-server\bin;$env:USERPROFILE\.config\herd-lite\bin;$env:Path"
```

ユーザー環境変数 `Path` に残っているPostgreSQL / Herd関連パスを確認する場合:

```powershell
[Environment]::GetEnvironmentVariable("Path", [EnvironmentVariableTarget]::User) -split ";" | Where-Object { $_ -match "PostgreSQL|herd-lite" }
```

旧PostgreSQLパスを削除する場合:

```powershell
$remove = "$env:LOCALAPPDATA\PostgreSQL\18.3\bin"
$userPath = [Environment]::GetEnvironmentVariable("Path", [EnvironmentVariableTarget]::User)
$parts = $userPath -split ";" | Where-Object { $_ -and ($_.TrimEnd('\') -ne $remove.TrimEnd('\')) }
[Environment]::SetEnvironmentVariable("Path", ($parts -join ";"), [EnvironmentVariableTarget]::User)
```

## 5. 開発DB確認

PostgreSQLサービス状態:

```powershell
Get-Service postgresql-x64-18
```

DB接続確認:

```powershell
$pgBin = "$env:LOCALAPPDATA\PostgreSQL\18-server\bin"
$env:PGPASSWORD = "jasuppliers_dev_password"
& "$pgBin\psql.exe" -w "host=127.0.0.1 port=5432 dbname=jasuppliers_dev user=jasuppliers_user connect_timeout=3" -c "SELECT current_database(), current_user;"
```

DBが未作成の場合:

```powershell
$pgBin = "$env:LOCALAPPDATA\PostgreSQL\18-server\bin"
$env:PGPASSWORD = "<PostgreSQLインストール時に設定したpostgres管理者パスワード>"
$env:PGCONNECT_TIMEOUT = "3"

& "$pgBin\psql.exe" -w "host=127.0.0.1 port=5432 dbname=postgres user=postgres connect_timeout=3" -v ON_ERROR_STOP=1 -c "CREATE USER jasuppliers_user WITH PASSWORD 'jasuppliers_dev_password';"
& "$pgBin\createdb.exe" -w -h 127.0.0.1 -p 5432 -U postgres -O jasuppliers_user jasuppliers_dev
```

`CREATE USER` が「既に存在する」で失敗する場合は、ユーザー作成済みのため `createdb` のみ実行する。

`.env` はGit管理しない。
上記DBパスワードはローカル開発用であり、共有環境や本番環境では環境ごとに変更する。

## 8. トラブルシュート

### 8.1 `php` / `composer` が見つからない

PowerShellを開き直す。

それでも見つからない場合:

```powershell
$env:Path = "$env:USERPROFILE\.config\herd-lite\bin;$env:Path"
php -v
composer --version
```

### 8.2 `psql` が止まる

パスワード待ちで止まっている可能性が高い。

必ず以下の形式にする。

```powershell
$pgBin = "$env:LOCALAPPDATA\PostgreSQL\18-server\bin"
$env:PGPASSWORD = "jasuppliers_dev_password"
& "$pgBin\psql.exe" -w "host=127.0.0.1 port=5432 dbname=jasuppliers_dev user=jasuppliers_user connect_timeout=3" -c "SELECT 1;"
```

### 8.3 PostgreSQLに接続できない

サービスを確認する。

```powershell
Get-Service postgresql-x64-18
```

起動していない場合:

```powershell
Start-Service postgresql-x64-18
```

### 8.4 `pg_ctl` が残っている

前回の中断により `pg_ctl` プロセスが残る場合がある。

PostgreSQLサービス `postgresql-x64-18` が `Running` であり、`psql` 接続確認が通れば開発作業には影響しない。

確認:

```powershell
Get-Process psql,pg_ctl -ErrorAction SilentlyContinue
```

`psql` が残っている場合は、パスワード入力待ちで停止した可能性がある。プロセスIDを確認してから停止する。

```powershell
Stop-Process -Id <対象psqlのID>
```

`pg_ctl` のみが残る場合は、サービス管理下または前回中断時の残存プロセスである可能性がある。DB接続が成功していれば作業継続可能であり、気になる場合はWindows再起動で解消する。

通常権限で `pg_ctl` の詳細確認がアクセス拒否になる場合がある。PostgreSQLサービスとDB接続が正常であれば、管理者権限での強制停止は不要とする。
